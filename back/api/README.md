# REST API Structure (EN)

I divided the backend into **4 distinct folders**, each handling a specific part of the process. Inside each of these, there will be multiple files, like **"tableFolder"**, where:
 - **table** represents the name of the database table being worked with
 - **Folder** represents the type of actions that will be found there:

> To see all the routes, directly managed by the Router, without seeing the JSON/Status sending part, go to [Routes](../routes)

> To see how the entire logic part actually works, go to [Services](../services)

> To see all the routes, including the JSON/status sending part for each endpoint, go to [Controllers](../controllers)

> To see how the entire data management and retrieval from the databases works, go to [Repositories](../repositories)

# Structura REST API (RO)
Am impartit backend-ul in **4 foldere distincte**, fiecare ocupandu-se cu o anumita parte a procesului. In fiecare dintre acestea, se vor gasi mai multe fisiere, de forma **"tableFolder"**, unde:
 - **table** reprezinta numele tabelei din baza de date cu care se lucreaza
 - **Folder** reprezinta tipul de actiuni care se vor regasi acolo:

> Pentru a vedea toate rutele, direct gestionate de catre Router, fara a vedea partea de trimitere JSON/Status, acceseaza [Routes](../routes)

> Pentru a vedea efectiv cum functioneaza toata partea de logica, acceseaza [Services](../services)

> Pentru a vedea toate rutele, inclusiv partea de trimitere JSON/status pentru fiecare endpoint, acceseaza [Controllers](../controllers)

> Pentru a vedea cum functoneaza toata partea de gestionare si preluare date din bazele de date, acceseaza [Repositories](../repositories)
